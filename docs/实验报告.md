## 1. 实验目的

验证在 Windows PE 格式下，能否构造一种符合以下定义的 **第三种 DLL**：

> 外壳是合法 PE 动态链接库；**代码节（`.text`）主体为明文 Brainfuck**；运行时由极小原生引导层 + VM 解释执行；业务逻辑不以机器码形式存在。

实验需同时排除两类不合格产物：

| 类型 | 定义 |
|------|------|
| **欺骗 DLL** | 将 BF 预编译/JIT 为 x64/x86 机器码作为业务逻辑 |
| **伪装 DLL** | 将 BF 藏在 `.rsrc`、`.rdata` 或外部文件，退化为数据附属物 |

详细目标定义见 [可行性报告.md](./可行性报告.md) §1。

---

## 2. 实验结论

**结论：实验目标全部达成。**

已成功产出 `bfdll.dll`（x64 / MSVC），并通过自动化脚本、C/C# 宿主测试及人工抽查验证。该 DLL 可被 `LoadLibrary` / 导入库正常加载；BF 明文物理位于 `.text` 节；业务逻辑由 VM 逐字节解释 `.text` 内 ASCII 指令流执行，不构成欺骗或伪装 DLL。

## 4. 架构概要

```
宿主 (C / C# / LoadLibrary)
    ↓ 调用导出函数
Native Stub (.text 头部，极小块)
    ↓ bf_vm_run(code)
VM 解释循环 (C 实现，switch 读 +-<>[],.)
    ↓ 顺序读取
BF 明文 (.text 主体，MASM db 嵌入)
    ↓ '.' 指令
bf_io → 内部 buffer / BF_SetOutputCallback
```

**关键设计决策：**

- BF 仅经 **MASM `db`** 嵌入 `.text`，禁止 C 字符串常量（避免落入 `.rdata`）
- 段名 `bf_text` 链接时 `/MERGE:bf_text=.text` 合并进代码节
- 导出表只指向 Native Stub，CPU 不 jump 到 BF 字节
- 带 `; bfdll: export=output` 的 `.bf` 由 `bf2asm.py` 自动生成 `BF_<Name>()` 导出

## 8. 已知限制

| 限制 | 说明 |
|------|------|
| 架构 | 仅 x64，x86 未验证 |
| 输入 `,` | `bf_io_read()` 固定返回 0，交互式 BF 不可用 |
| 输出 buffer | 4KB 上限，超出截断 |
| 工具链 | 无独立 `bf2dll foo.bf` CLI，需 CMake 工程流程 |
| 参数导出 | 仅 `export=output` 无参自动导出；带参需手写 Stub（如 `BF_Add`） |
| 数字签名 | 本地构建无 Authenticode 签名 |

## 9. 复现步骤

```powershell
# 构建
cmake -S . -B build -G "Visual Studio 17 2022" -A x64
cmake --build build --config Release

# C 验收
cd build/bin/Release
.\test_host.exe
.\test_host_static.exe

# PE/BF 验收
powershell -ExecutionPolicy Bypass -File ..\..\..\tools\verify_dll.ps1 `
  -DllPath (Resolve-Path bfdll.dll)

# C# 验收
cd examples/BfdllCSharp
dotnet run -c Release
```

## 10. 总结

本实验在 Windows x64 上成功实现了 **Brainfuck-in-PE 第三种 DLL**：BF 明文占据 PE 代码节主体，Windows 加载器将其识别为合法 DLL，业务语义完全由 VM 对 `.text` 内 ASCII 指令流的解释执行完成。实验排除了「欺骗」与「伪装」两类不合格形态，并通过脚本、多语言宿主与人工记事本搜索交叉验证。
