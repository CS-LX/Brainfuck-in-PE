# BFDll 实现方案

> 对应文档：[可行性报告.md](./可行性报告.md)  
> 日期：2026-06-25  
> 目标：产出可被正常调用的 `xxx.dll`，且 `.text` 主体为明文 Brainfuck

---

## 1. 设计总览

### 1.1 核心思路

把 PE 的 `.text` 节重新定义为两个逻辑区域，物理上仍在同一 `IMAGE_SCN_MEM_EXECUTE` 节内：

```
.text（单一 PE 节，属性 R+X）
┌──────────────────────────────────────────────────────────┐
│ Region A │ Native 引导层（Stub + VM）     ~2–8 KB       │
│          │ 导出表 EAT 只指向这里                         │
├──────────┼───────────────────────────────────────────────┤
│ Region B │ BF 明文指令流（业务逻辑 100%）  占 .text 90%+ │
│          │ CPU 永不 jump 到此；VM 以 const char* 顺序读  │
└──────────────────────────────────────────────────────────┘
```

**运行时数据流：**

```
宿主调用 BF_Add(3, 5)
    → EAT 跳转到 Stub（Region A）
    → Stub 把 RCX/RDX 写入 tape[0..1]
    → VM 从 Region B 的 BF_Prog_Add 处取指令解释
    → 结束后 Stub 读 tape[0] → RAX → ret
```

### 1.2 设计原则

| 原则 | 做法 |
|------|------|
| BF 是程序本体 | 所有业务逻辑只写在 `.bf` 源文件，经工具链嵌入 `.text` |
| Stub 是胶水 | Native 代码只做 ABI、VM 调度、I/O 钩子，不含业务算法 |
| 禁止编译 BF | 无 AOT、无 JIT、无 BF→IL；VM 循环是唯一执行路径 |
| BF 必须在代码节 | 用 MASM `CODE` 段 + `db` 嵌入，不用 `.rsrc` / `.rdata` |
| **禁止 C 字符串嵌入** | BF **不得**以 C/C++ 字符串字面量、`const char[]`、`#embed` 等任何形式出现在 C 源文件中（见 §3.1） |
| **记事本可见** | BF 以 ASCII 明文写入 `.dll` 磁盘镜像；用记事本打开可搜索到 `+-<>[],.`（见 §8.3） |
| 可验证 | 提供 `dumpbin` / 脚本 / 记事本搜索验收，量化 `.text` 中 ASCII BF 占比 |

### 1.3 技术选型

| 组件 | 选择 | 理由 |
|------|------|------|
| 语言 | C11（VM/Stub）+ MASM（BF 嵌入） | 可控、无异常、MSVC 官方工具链 |
| 构建 | CMake + MSVC v143 | `#pragma code_seg`、`.def`、`dumpbin` 一站式 |
| 架构 | 先 x64，后 x86 | x64 默认 `__cdecl`，与 Windows API 一致 |
| BF 嵌入 | `bf2asm.py` → `.asm` → `ml64` | **唯一合法路径**；C 字符串会被 MSVC 放进 `.rdata`，属伪装 |
| 宿主测试 | 纯 C `LoadLibrary` 小程序 | 不依赖 .NET，验证真实 DLL 加载路径 |

## 3. PE 布局：如何把 BF 放进「放指令的地方」

### 3.1 为何用汇编而非 C 字符串（硬性禁止）

MSVC 中即使用 `#pragma code_seg(".text$bf")`，`const char s[] = "+++"` 仍可能被放进 `.rdata`（只读常量池）。一旦 BF 落入 `.rdata`，即构成**伪装 DLL**（BF 退化为数据节附属物，且记事本虽可能搜到，但不在「放指令的地方」）。

**BF 嵌入唯一合法路径：MASM 在 `CODE` 段内用 `db` 逐字节写入 BF 字符。**

#### 禁止清单（C/C++ 源文件中不得出现 BF 程序字节）

以下写法**一律禁止**用于存放 Brainfuck 程序本体；Code Review 与 `verify_dll.ps1` 均视为违规：

| 禁止写法 | 为何违规 |
|----------|----------|
| `const char g_bf[] = "[->+<]";` | 常量池 → `.rdata` |
| `#pragma code_seg(".text$bf")` + 字符串字面量 | MSVC 仍可能下沉至 `.rdata` |
| `char g_bf[] = { '[', '-', '>', ... };` 在 `.c` 中 | 初始化数据 → `.data` / `.rdata` |
| `#embed "add.bf"` / `#include "add.bf"` 进 C 源 | 本质仍是 C 编译管线，节区不可控 |
| 运行时 `ReadFile("add.bf")` | BF 不在 DLL 体内 |
| RC/`.rc` 资源、`TEXTINCLUDE` | 资源区嵌入 → 伪装 |

Stub 中只允许通过 **链接器符号** 引用 BF，例如：

```c
// bf_stub.c — 仅声明，定义在 MASM 生成的 .obj 中
extern const char BF_Prog_Add[];   // 实际由 BF_Prog_Add LABEL BYTE + db 提供

bf_vm_run(&vm, BF_Prog_Add);       // OK：指针指向 .text 内 BF 明文
```

**禁止**在 C 文件中写 `BF_Prog_Add` 的初值或字符串内容。

```asm
; src/asm/bf_programs.asm（示意）
_TEXT$bf SEGMENT PARA PUBLIC 'CODE'
    ALIGN 16

    PUBLIC BF_Prog_Hello
BF_Prog_Hello LABEL BYTE
    db '+++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++.', 0

    PUBLIC BF_Prog_Add
BF_Prog_Add LABEL BYTE
    ; cell0=a, cell1=b → cell0=a+b（8-bit wrap）
    db '[->+<]', 0

_TEXT$bf ENDS
END
```

链接后 `.text$bf` 自动合并进 `.text`（MSVC 默认行为）。`BF_Prog_Add` 符号即 Region B 中的入口偏移。

### 3.2 Region A 的段排序

利用 MSVC `$` 段名字母序合并规则，保证 Stub 在 BF 之前（导出函数低地址、反汇编先看到合法机器码）：

| 段名 | 内容 |
|------|------|
| `.text$a` | DllMain、导出 Thunk（`bf_stub.c` 用 `#pragma code_seg(".text$a")`） |
| `.text$m` | VM 解释器（`bf_vm.c` → `.text$m`） |
| `.text$bf` | BF 明文（MASM 生成） |

Stub 体积小且集中在低地址；BF 块紧随其后，占据 `.text` 剩余大部分空间。

### 3.3 链接器选项

```cmake
target_link_options(${target} PRIVATE
    "/DEF:${CMAKE_SOURCE_DIR}/exports/bfdll.def"
    "/SUBSYSTEM:WINDOWS"           # DLL 无控制台
    # 确保 BF 符号不被优化掉
    "/INCLUDE:BF_Prog_Hello"
)
```

不使用 `/MERGE` 把 BF 从别处合并——BF 天生就在 `.text$bf` 里。

### 3.4 内存布局示意（链接后）

```
RVA          内容                          dumpbin / 十六进制
─────────────────────────────────────────────────────────────
0x1000       DllMain, BF_Add, BF_Hello...  正常 x64 反汇编
0x1200       bf_vm_run 解释循环             正常 x64 反汇编
0x1800       BF_Prog_Hello: 2B 2B 2B...    ASCII '+++...'
0x1900       BF_Prog_Add:   5B 2D 3E...    ASCII '[->+<]'
...
```

验收标准：`.text` 总大小中，**非 Stub/VM 的 ASCII BF 字节占比 > 50%**（MVP 后随程序增大趋近 90%）。

## 4. 模块设计

### 4.1 VM 解释器（`bf_vm.c`）

**职责：** 唯一执行 BF 的组件；从 `.text` 内指针读取指令；不生成、不跳转至任何动态机器码。

```c
// bf_vm.h（接口摘要）
typedef struct {
    uint8_t* tape;
    size_t   tape_size;      // 默认 65536
    size_t   tape_ptr;
    uint64_t step_limit;     // 防死循环，默认 10_000_000
    void*    io_user;        // I/O 回调上下文
} bf_vm_t;

typedef enum {
    BF_OK = 0,
    BF_ERR_UNMATCHED_BRACKET,
    BF_ERR_STEP_LIMIT,
    BF_ERR_IO,
} bf_status_t;

bf_status_t bf_vm_run(bf_vm_t* vm, const char* code);
// code 必须指向 .text 内 BF 明文；VM 只读，不修改 code
```

**解释循环（伪代码）：**

```
while (1) {
    switch (*ip) {
    case '+': tape[ptr]++; ip++; break;
    case '-': tape[ptr]--; ip++; break;
    case '>': ptr++; ip++; break;
    case '<': ptr--; ip++; break;
    case '.': bf_io_write(vm, tape[ptr]); ip++; break;
    case ',': tape[ptr] = bf_io_read(vm); ip++; break;
    case '[': if (tape[ptr]==0) ip = jump_table[ip]; else ip++; break;
    case ']': if (tape[ptr]!=0) ip = jump_table[ip]; else ip++; break;
    case '\0': return BF_OK;
    default:   ip++; break;   // 忽略空白/注释（可选）
    }
    if (++steps > limit) return BF_ERR_STEP_LIMIT;
}
```

**括号跳转优化：** 首次执行某 `code` 地址时，扫描一遍构建 `jump_map`（可用小型 LRU 缓存 keyed by `code` 指针）。缓存的是 **IP 跳转偏移**，不是 native 机器码；BF 在 `.text` 只读，缓存可复用。

**明确禁止：**

- `VirtualAlloc` + 写入 op 对应机器码
- `switch` 内联汇编生成指令
- 任何将 BF 翻译为函数指针并调用的路径

### 4.2 I/O 钩子（`bf_io.c`）

| 指令 | MVP 行为 | 后续扩展 |
|------|----------|----------|
| `.` | 单字符写入 thread-local 缓冲区；Stub 返回后可读 | 回调 `bf_write_fn` |
| `,` | 始终读 `0` | 从 `tape[2..]` 预置输入流读取 |

`hello.bf` 通过多次 `.` 输出字符串，Stub 在 `BF_Hello` 返回前把 buffer flush 到 `OutputDebugStringA`。

### 4.3 ABI 桥接 — BFC-0 约定

实验期固定一套「Brainfuck Calling Convention 0」，写入 `bf_abi.h`：

```
BFC-0（x64 Windows）
────────────────────
调用约定：__cdecl
纸带：    uint8_t[BFDLL_TAPE_SIZE]，默认 65536，环绕
参数：    第 i 个整型参数 → tape[i]（截断至 uint8_t）
返回值：  执行结束后 tape[0] 作为 int 返回（零扩展至 32/64 bit）
终止：    IP 遇到 '\0'
错误：    VM 返回非 BF_OK 时，Stub 返回 -1（或 SetLastError，二期）
```

**Stub 模板（每个导出函数一份，极薄）：**

```c
#pragma code_seg(".text$a")

__declspec(dllexport) int __cdecl BF_Add(int a, int b)
{
    uint8_t tape[BFDLL_TAPE_SIZE] = {0};
    bf_vm_t vm = bf_vm_default(tape);
    tape[0] = (uint8_t)a;
    tape[1] = (uint8_t)b;

    bf_status_t st = bf_vm_run(&vm, BF_Prog_Add);  // 符号来自 .asm
    return (st == BF_OK) ? (int)tape[0] : -1;
}
```

**多导出函数：** 每个导出对应独立 BF 块 + 独立 Stub；`.def` 控制导出名：

```def
LIBRARY bfdll
EXPORTS
    BF_Hello
    BF_Add
```

### 4.4 DllMain（`dllmain.c`）

```c
BOOL WINAPI DllMain(HINSTANCE h, DWORD reason, LPVOID reserved)
{
    (void)h; (void)reserved;
    switch (reason) {
    case DLL_PROCESS_ATTACH:
        bf_io_init();          // 仅重置 TLS buffer
        break;
    case DLL_PROCESS_DETACH:
        bf_io_shutdown();
        break;
    }
    return TRUE;
}
```

不在 attach 时预跑 VM、不分配大内存、不调用 `LoadLibrary`。

## 5. 工具链：`bf2asm.py`

### 5.1 输入 / 输出

```
输入：src/programs/add.bf
输出：src/asm/gen/add.asm（或合并进 bf_text.inc）
```

### 5.2 生成规则

1.  strip 非 `+-<>[],.` 字符（保留换行便于阅读，输出时去掉）
2.  每行最多 64 个 `db 'xxx'` 字符，避免 MASM 行过长
3.  末尾 `db 0` 作为 BFC-0 终止符
4.  生成 `PUBLIC BF_Prog_<Name>` 标签
5.  可选：生成 `.bf.meta` 记录原始 SHA256，供 verify 脚本对比

**生成示例：**

```asm
; AUTO-GENERATED from add.bf — DO NOT EDIT
_TEXT$bf SEGMENT PARA PUBLIC 'CODE'
    PUBLIC BF_Prog_Add
BF_Prog_Add LABEL BYTE
    db '[->+<]', 0
_TEXT$bf ENDS
```

### 5.3 CMake 集成

```cmake
function(add_bfdll TARGET)
    # 1. 对每个 .bf 运行 bf2asm.py
    # 2. ml64 /c gen/*.asm
    # 3. 链接 vm + runtime + asm objs → ${TARGET}.dll
endfunction()

add_bfdll(bfdll)
add_executable(test_host host/test_host.c)
target_link_libraries(test_host PRIVATE bfdll)  # 可选：通过 import lib
```

构建顺序：

```
programs/*.bf
    → bf2asm.py
    → gen/*.asm
    → ml64
    → *.obj  ─┐
              ├→ link → bfdll.dll + bfdll.lib
vm/*.c ───────┤
runtime/*.c ──┘
```

// host/test_host.c
#include <windows.h>

typedef int (__cdecl *BF_Add_fn)(int, int);

__declspec(dllimport) int __cdecl BF_Add(int, int);

## 8. 验收方案

### 8.1 自动脚本 `verify_dll.ps1`

| 检查项 | 方法 | 通过条件 |
|--------|------|----------|
| 是 DLL | `dumpbin /headers` | 含 `DLL` 特征 |
| 有导出 | `dumpbin /exports` | 列出 `BF_Add` 等 |
| BF 在 .text | 解析 PE Section Table | `.text` VirtualSize 最大 |
| **BF 不在 .rdata** | 扫描 `.rdata` RawData | 不含完整 BF 程序串（如 `[->+<]`、`programs/*.bf` 的 SHA256 片段） |
| 无 BF 资源 | `dumpbin /summary` | 无 `.rsrc` 或 `.rsrc` < 512B |
| 明文可见（二进制） | 读取 `.text` RawData | 存在连续 `+-<>[],.` 序列 ≥ 16 字节 |
| **记事本可见** | 见 §8.3 | 对 `.dll` 全文 UTF-8/ASCII 搜索每个 `.bf` 源的核心片段，全部命中 |
| 非机器码主体 | 统计 `.text` 熵 / ASCII 比 | BF 区域 ASCII 可打印字符 > 80% |
| 无 C 字符串嵌入 | 构建日志 / 静态 grep | `src/` 下 `.c/.h` 无 BF 指令字符字面量（`[->+<]` 等） |
| 功能 | 运行 `test_host.exe` | 加法结果正确 |

### 8.2 人工抽查

1. 用十六进制编辑器打开 `bfdll.dll`，跳转到 `.text` 文件偏移。
2. 在 Stub 机器码之后应看到 `5B 2D 3E 2B 3C 5D`（`[->+<]`）。
3. IDA 反汇编 `.text`：Stub 正常；BF 区域显示为 `db` 式乱码——**符合预期**。

### 8.3 记事本可见 BF 明文（硬性验收）

**目的：** 证明 BF 以 ASCII 明文写入 `.dll` 磁盘文件，而非仅存在于内存或工具解析结果中。

**说明：** 用记事本双击打开 `.dll` 时，PE 头与 Stub 机器码会显示为乱码，**整文件不会像 `.txt` 一样全可读**——这是正常现象。验收方式是 **在记事本内 Ctrl+F 搜索 BF 源中的特征片段**，或脚本等价扫描整个文件字节流。

| 步骤 | 操作 | 通过条件 |
|------|------|----------|
| 1 | 资源管理器中双击 `bfdll.dll`，用记事本打开 | 文件能打开（二进制混排视图） |
| 2 | Ctrl+F 搜索 `add.bf` 的核心逻辑 `[->+<]` | **必须找到** 连续 ASCII 匹配 |
| 3 | Ctrl+F 搜索 `hello.bf` 中的长 `+` 串或特征子串 | **必须找到**（长度 ≥ 8 的 BF 字符序列） |
| 4 | 确认匹配位置不在 `.rsrc` 区 | 结合 `dumpbin`：匹配偏移落在 `.text` 的 FileOffset 范围内 |

**脚本等价检查（纳入 `verify_dll.ps1`）：**

```powershell
$dll = Get-Content -Path $DllPath -Encoding Byte -Raw
$text = [System.Text.Encoding]::ASCII.GetString($dll)
# 每个 programs/*.bf 经 bf2asm  strip 后的核心串都必须命中
if ($text -notmatch '\[->\+<\]') { throw 'Notepad test failed: add.bf fragment not in DLL' }
```

**与实验目标的关系：**

- 满足「真的把 BF 明文写进 dll 文件」——磁盘上存在可搜索的 `+-<>[],.` ASCII 字节；
- 若 BF 仅存在于 `.rdata` / `.rsrc`，记事本可能搜到，但 §8.1「BF 不在 .rdata」与「BF 在 .text」会同时判失败；
- 若 BF 被编译为机器码，记事本搜不到 `[->+<]` 等片段，本项判失败。

### 8.4 对照实验（证明不是欺骗/伪装）

| 对照组 | 预期失败哪条 |
|--------|--------------|
| BF 放 `.rsrc` | 目标 4、5 |
| BF AOT 编译为机器码 | 目标 3 |
| BF 放 `.rdata` 字符串 | 目标 5 |
| **本方案** | 全部通过 |

## 10. 关键风险与对策

| 风险 | 对策 |
|------|------|
| 链接器 GC 掉 BF 符号 | `/INCLUDE:BF_Prog_*` + `PUBLIC` |
| BF 被误当代码执行 | 导出/入口只指向 Region A；EAT 无 BF 内偏移 |
| **C 字符串悄悄进 `.rdata`** | §3.1 禁止清单 + grep 检查 + `verify_dll.ps1` 扫 `.rdata` |
| **记事本搜不到 BF** | 说明未真正落盘或 strip 过度；对照 `programs/*.bf` 与 `gen/*.asm` |
| `.text` 内 BF 仍带 X 属性 | 可接受；CPU 不 jump 即不触发；实验不改为 RW 节（否则失去「代码节」语义） |
| 8-bit cell 溢出 | BFC-0 文档声明；二期可扩展 BFC-1 用 32-bit cell |
| 死循环 | `step_limit` 强制中断 |

## 11. 实现思路一句话

> **用 MASM 把 BF 明文钉死在 `.text` 节主体；用极薄 C Stub 接 Windows ABI；用纯解释 VM 读 `.text` 内的 BF 完成业务——磁盘上代码区是 Brainfuck，加载器眼里仍是标准 DLL。**
